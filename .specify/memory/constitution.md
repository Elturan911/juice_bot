# juice_bot Constitution

## Основные принципы

### I. Простота и синхронность

Проект строго отдаёт предпочтение простому, понятному синхронному коду перед
сложными асинхронными решениями.

- Асинхронные паттерны (async/await, threading, multiprocessing) ЗАПРЕЩЕНЫ, если
  не доказано и не задокументировано узкое место производительности.
- Каждый новый код ДОЛЖЕН быть прочитан разработчиком-джуном без затруднений.
- Сложность ДОЛЖНА быть обоснована в разделе Complexity Tracking плана.

**Почему**: Бот обрабатывает единичные сообщения последовательно; накладные расходы
асинхронности не окупаются, а усложняют поддержку.

### II. LLM-первичный разбор

Всё понимание естественного языка — парсинг трат, категоризация, извлечение сумм —
ДОЛЖНО выполняться через LLM со структурированным JSON-выводом.

- Регулярные выражения и rule-based парсеры для NLU ЗАПРЕЩЕНЫ.
- Каждый LLM-вызов ДОЛЖЕН использовать Pydantic-схему для валидации ответа.
- Промпты ДОЛЖНЫ включать примеры (few-shot) для стабильности вывода.

**Почему**: Хрупкие regex-парсеры ломаются при малейших изменениях пользовательского
ввода; LLM с JSON-схемой даёт предсказуемый и типобезопасный результат.

### III. Обязательный стек технологий

Отклонения от указанного стека ЗАПРЕЩЕНЫ без поправки к конституции.

| Слой | Технология |
|---|---|
| Язык | Python 3.11+ |
| Менеджер зависимостей | Poetry |
| Telegram SDK | python-telegram-bot v20 (синхронный режим) |
| ORM | SQLAlchemy (sync mode) |
| Драйвер БД | psycopg2-binary |
| LLM | Claude (Anthropic API) |
| Валидация схем | Pydantic v2 |
| Линтер | ruff |
| Форматтер | black + isort |

**Почему**: Единый стек снижает когнитивную нагрузку, упрощает онбординг и позволяет
применять заготовленные шаблоны.

### IV. Строгое качество кода

- `black` и `isort` ДОЛЖНЫ запускаться автоматически (pre-commit hook или CI).
- `ruff` не ДОЛЖЕН выдавать ни одной ошибки в итоговом коде.
- Весь код ДОЛЖЕН проходить линтинг перед коммитом.
- Комментарии пишутся на русском языке; код (имена переменных, функций) — на
  английском.

**Почему**: Автоматическое форматирование исключает споры о стиле и снижает нагрузку
на ревью.

### V. Безопасность и управление секретами

- API-ключи, строки подключения к БД и любые credentials ЗАПРЕЩЕНО хранить в
  исходном коде или в git-истории.
- Для локальной разработки: переменные окружения в `.env` (файл в `.gitignore`).
- Для продакшна: `fly secrets set` — единственный допустимый способ передачи секретов.
- `.env.example` ДОЛЖЕН существовать с заглушками для всех необходимых переменных.

**Почему**: Утечка credentials в публичный репозиторий необратима и критична.

### VI. Обязательная проверка документации через Context7

**Это правило ОБЯЗАТЕЛЬНО и не может быть пропущено ни при каких обстоятельствах.**

Перед составлением любого плана (`/speckit.plan`) и перед выполнением любой задачи
разработчик (или агент) ОБЯЗАН воспользоваться инструментом Context7 MCP для получения
актуальной документации по всем библиотекам, которые будут использоваться.

- Context7 ДОЛЖЕН вызываться даже если библиотека хорошо известна — версии и API
  меняются, и внутренние знания могут быть устаревшими.
- В плане (`plan.md`) в разделе "Technical Context" ОБЯЗАН быть указан результат
  проверки документации (версия библиотеки, подтверждённый API).
- Пропуск шага Context7 является нарушением конституции и блокирует продвижение задачи.

**Библиотеки, которые ВСЕГДА требуют проверки через Context7**:
`python-telegram-bot`, `SQLAlchemy`, `anthropic` (Anthropic SDK), `Pydantic`,
`alembic`, `psycopg2`, `Poetry`, `fly` CLI.

**Почему**: Даже хорошо известные библиотеки регулярно меняют API между minor-версиями.
Использование устаревшей документации — источник трудноуловимых багов и потери времени.

### VII. Экономная инфраструктура

- Ежемесячные расходы на хостинг ДОЛЖНЫ оставаться ниже $5.
- Инфраструктура ОБЯЗАНА масштабироваться до нуля при отсутствии трафика.
- Допустимые платформы: Fly.io (compute) + Neon (PostgreSQL).
- Любое решение, превышающее бюджет, ДОЛЖНО быть обосновано и одобрено.

**Почему**: Проект является учебным/личным; экономия ресурсов — ключевое ограничение.

## Тестирование и надёжность

- Критическая бизнес-логика (финансовые расчёты, запросы к БД) ДОЛЖНА быть покрыта
  тестами, которые выполняются перед деплоем.
- Тесты ДОЛЖНЫ запускаться против реальной (тестовой) БД, а не моков, чтобы
  исключить расхождение контрактов.
- Для некритических вспомогательных функций тесты ОПЦИОНАЛЬНЫ.
- Команда запуска тестов: `pytest` из директории `src/`.

## Пользовательский опыт

- Бот ДОЛЖЕН направлять пользователя понятными сообщениями на каждом шаге.
- Ошибки ДОЛЖНЫ обрабатываться gracefully — пользователь НИКОГДА не должен видеть
  stack trace или техническое сообщение об ошибке.
- UX-поток ДОЛЖЕН быть проверен вручную через реальный Telegram до закрытия задачи.
- Все тексты интерфейса — на русском языке.

## Governance

- Конституция является высшим документом проекта и имеет приоритет над всеми другими
  практиками и договорённостями.
- Любое отклонение от принципов ДОЛЖНО быть оформлено как поправка к конституции с
  обновлением версии и датой.
- Процедура поправки:
  1. Описать изменение и обоснование.
  2. Обновить версию по semver (MAJOR/MINOR/PATCH).
  3. Обновить `LAST_AMENDED_DATE`.
  4. Проверить согласованность всех шаблонов (checklist в этом документе выше).
- Все задачи в `tasks.md` ДОЛЖНЫ содержать ссылку на принцип конституции, который
  они реализуют.
- Шлюз перед Phase 0: Constitution Check ДОЛЖЕН быть пройден явно в `plan.md`.
- Для разработки используй Context7 MCP для получения актуальной документации (Принцип VI).
